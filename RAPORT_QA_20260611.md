## Trzy błędy typowe dla podobnych aplikacji

1. **REVOKE/GRANT mylone z RLS** → sprawdzono migracje → ❌ POPEŁNIONY (migracja 22) — naprawiony migracją 25. RLS filtruje wiersze, GRANT-y dają dostęp do tabeli; potrzebne są OBA.
2. **Redirecty budowane z `request.url` za reverse proxy** → sprawdzono signout/callback/middleware → ❌ POPEŁNIONY — naprawiony `resolveOrigin()` (x-forwarded-host).
3. **Deploy z niewłaściwej gałęzi (main vs master)** → sprawdzono VPS → ❌ POPEŁNIONY — VPS klonował `main` (stub), prawdziwy kod żył na `master`. Naprawione przełączeniem. **Rekomendacja: usunąć gałąź `main` z GitHub** albo zrobić z niej mirror master, żeby to się nie powtórzyło.

## Jedno inteligentne usprawnienie QA

**Smoke test po każdym deployu** — skrypt `scripts/smoke.sh` na VPS uruchamiany po `pm2 restart`:
```bash
#!/bin/bash
fail=0
for path in / /login /register /kalkulator /flota /forgot-password /onboarding/driver; do
  code=$(curl -s -o /dev/null -w "%{http_code}" "http://localhost:3000$path")
  [[ "$code" =~ ^(200|307)$ ]] || { echo "FAIL $path -> $code"; fail=1; }
done
curl -s http://localhost:3000/kalkulator | grep -q 'min="20" max="200"' || { echo "FAIL: stary kalkulator!"; fail=1; }
curl -s -X OPTIONS -H "Origin: https://adride.pl" -o /dev/null -w "%{http_code}\n" http://localhost:3000/api/leads | grep -q 204 || { echo "FAIL: CORS leads"; fail=1; }
exit $fail
```
Ten jeden skrypt wykryłby dziś **wszystkie trzy** błędy krytyczne przed użytkownikiem.

## Pytanie produktowe (zgłoszone przez Ciebie)

> „Nie powinno być tak, że nowy kierowca najpierw wypełnia formularz, a potem ma login?"

Obecny flow ma już obie ścieżki: **(a)** landing kierowca.html = formularz leadowy BEZ konta (zbierasz kontakt, dzwonisz, kwalifikujesz auto) i **(b)** app = konto → onboarding (dane do umowy i wypłat). To dobry układ dla sprintu sprzedażowego — lead przed kontem, konto dopiero gdy kierowca przechodzi do umowy. Rekomendacja: zostawić, ale na stronie sukcesu leada dodać link „Załóż konto teraz" dla niecierpliwych.

### Dodatkowy bug naprawiony podczas deployu — INFRA-01 (nginx 521)
Po `pm2 restart` Cloudflare zwracał **521 "Web server is down"**. PM2 był online, `curl localhost:3000` działał — winowajcą był nginx, który nie wstał przez osierocony plik `/etc/nginx/sites-available/ja-adride-log` wskazujący na nieistniejący certyfikat `ja.adride.pl`. Fix: `rm` osieroconego pliku + `systemctl restart nginx`. **Lekcja:** przy 521 zawsze `sudo nginx -t`; cert istnieje tylko dla `app.adride.pl`. Udokumentowane w `DEPLOYMENT.md` (Troubleshooting).
